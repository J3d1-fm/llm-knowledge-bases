LLM Knowledge Bases Technical Documentation

Version: v0.5.0
Date: 2026-06-15

Purpose
This repository contains a dependency-free static frontend for presenting the LLM Knowledge Bases product concept, a local markdown vault used as the knowledge source of truth, and an authenticated workspace backed by Firebase Auth and Firestore.

- styles.css: Responsive styling and visual system.

- firebase-config.js: Browser-side Firebase Web App configuration and allowed owner email.
- firebase.json: Firebase Hosting and Firestore deployment configuration.
- .firebaserc: Firebase project alias for llm-knowledge-bases.

Vault Authoring Model
- Edit vault markdown files first.
- Frontmatter stores structured fields that the app needs.
- Article bodies are extracted from the markdown body after the H1.
- Firestore is derived from the vault with npm run seed:firestore.
- The managed Firestore collections are articles, sources, checks, and outputs under vaults/main.
- The seed script deletes existing documents in those four collections before writing the current vault snapshot, preventing stale removed pages from staying visible.
- rawCount, articleCount, sourceCount, outputCount, and wordCount are derived from vault files during seed generation.
- Vault validation requires every raw markdown file to have a matching source record with the same ID.

CLI Access
The local CLI is the preferred agent entrypoint for inspecting the actual vault:
- npm run kb -- stats
- npm run kb -- search "query" --limit 10
- npm run kb -- show <id-or-path>
- npm run kb -- health
- npm run kb -- ingest <path> --dry-run
- npm run kb -- ingest <path>
- npm run kb -- register-raw

The ingest command imports .md and .txt files into vault/raw with generated frontmatter. register-raw creates missing source records for raw files. After importing a real corpus, run npm run validate before seeding Firestore.

Runtime
- No application server is required.
- No frontend package installation is required.
- The public site can be opened directly from index.html or served by any static host.
- app.html should be served from Firebase Hosting for Google sign-in and Firestore reads. Direct file opening is only for layout review.

Authentication Model
- The workspace uses Firebase Authentication with Google sign-in.
- The frontend also checks allowedEmails in firebase-config.js for UX clarity.
- The real access boundary is Firestore rules, which allow read access only for request.auth.token.email == "j3d1fm@gmail.com" and request.auth.token.email_verified == true.
- Client writes are disabled by rules.
- Private source imports should not be shipped in dist/ or committed to a public repository.

Auth Setup Blocker
- Firebase Hosting and Firestore are deployed.
- Firestore seed data is uploaded.
- Unauthenticated Firestore REST reads return 403.
- Firebase Authentication provider setup is still blocked outside code: Identity Toolkit returns CONFIGURATION_NOT_FOUND for accounts:createAuthUri.
- REST initialization through identityPlatform:initializeAuth returned BILLING_NOT_ENABLED.
- If avoiding billing, initialize Authentication manually in Firebase Console:
  Firebase Console -> llm-knowledge-bases -> Authentication -> Get started -> Sign-in method -> Google -> Enable -> Save.
- After this, verify app.html sign-in with j3d1fm@gmail.com.

Deployment
Primary deployment target is Firebase Hosting project llm-knowledge-bases. Build dist/ before deployment, then deploy Hosting and Firestore rules.

Commands:
node scripts/validate-static.mjs
node scripts/smoke-app.mjs
node scripts/validate-vault.mjs
node scripts/build-pages.mjs
npm run seed:firestore
firebase deploy --project llm-knowledge-bases --only hosting,firestore:rules

The dist artifact includes:
- index.html
- app.html
- styles.css
- script.js
- app.js
- firebase-config.js
- assets/

Rollback
The first releasable state is tagged as v0.1.0 in git. The first deploy-ready state is tagged as v0.1.1. The populated workspace release is tagged as v0.2.0. The markdown-vault source-of-truth release is tagged as v0.3.0. The first expanded content batch is tagged as v0.4.0. The local vault CLI release is intended to be tagged as v0.5.0. A rollback can restore any release tag.

Validation
Run:
node scripts/validate-static.mjs
node scripts/smoke-app.mjs
node scripts/validate-vault.mjs
node scripts/build-pages.mjs

The static validator checks that required files exist, local asset references resolve, and the page has the expected title and H1. The vault validator checks structured markdown data integrity before seeding Firestore. The build script creates the deployable dist/ artifact.
