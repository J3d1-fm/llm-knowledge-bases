Purpose
This repository contains a dependency-free static frontend for presenting the LLM Knowledge Bases product concept, a local markdown vault used as the knowledge source of truth, and an authenticated workspace backed by Firebase Auth and Firestore.

Vault Authoring Model
- Edit vault markdown files first.
- Frontmatter stores structured fields that the app needs.
- Article bodies are extracted from the markdown body after the H1.
- Firestore is derived from the vault and remote-safe workdb context with npm run seed:firestore.
- The managed Firestore collections are articles, sources, checks, outputs, and workdbContext under vaults/main.
- The seed script deletes existing documents in those five collections before writing the current vault and Work DB snapshots, preventing stale removed pages from staying visible.
- rawCount, articleCount, sourceCount, outputCount, and wordCount are derived from vault files during seed generation.
- Vault validation requires every raw markdown file to have a matching source record with the same ID.

CLI Access
The local CLI is the preferred agent entrypoint for inspecting the actual vault:
- npm run kb -- stats
- npm run kb -- search "query" --limit 10
- npm run kb -- show <id-or-path>
- npm run kb -- health
- npm run kb -- ingest <path> --dry-run
- npm run kb -- ingest <path>
- npm run kb -- register-raw

The ingest command imports .md and .txt files into vault/raw with generated frontmatter. register-raw creates missing source records for raw files. After importing a real corpus, run npm run validate before seeding Firestore.

Global Work DB
The global work database is generated under outputs/global-work-kb and is intentionally not committed. It contains:
- db.json: aggregate metadata, projects, sessions, external inventory, and tag counts.
- files.jsonl: file-level metadata and safe snippets for readable non-sensitive files.
- projects.jsonl: local project cards with git remotes when available.
- sessions.jsonl: Codex and Claude session/task metadata.
- external-inventory.json: optional GitHub, GCloud, and Firebase inventory.
- raw-registry.jsonl: generated raw diary/index snapshot with extracted metadata, safety flags, tags, and compile status.
- chronology.jsonl: generated chronological index across local files, Codex sessions, and Claude session/task files.
- catalog.md: inferred master catalog of current work clusters and CLI entry points.
- provenance.md: provenance policy for extracted, inferred, and ambiguous claims.
- tag-cloud.html: Obsidian/Hermes-style clustered memory graph with the agent memory lens, theme clusters, projects, raw samples, sessions, tags, repos, and cloud resources.
- analysis/: Generated markdown dossiers and JSON manifests for selected graph tags and clusters.
- remote-workdb-context.json: Firestore-ready remote-safe Work DB context documents for the authenticated app. It excludes local paths, snippets, file content, and git remotes.
- README.md: generated summary.

The raw source files are the diary-style layer. The raw registry and chronology are rebuildable generated indexes over that layer, not append-only audit logs. The catalog is the living interpretation layer and marks project/theme grouping as inferred. The graph is an operator view over those indexes, not the source of truth. The canvas renders a readable overview graph instead of drawing every indexed file/session as a node; full matched records are exposed through tag/cluster analysis manifests. The graph draws explicit theme clusters, hover magnification, pan/zoom, Fit all, explicit zoom out / zoom in controls, cluster focus controls, search focus, and a hidden-by-default inspector drawer. A connection-point layer marks every linked node endpoint, including theme-cluster centers. Edge paths now expose a single ordered vertex array used by both line drawing and bend-dot drawing; every internal path vertex is marked, including both bends on long links, so graph lines do not terminate or turn at invisible points. The graph has a low-cost motion layer that slowly rotates the rendered coordinates by about 2.6 degrees without recalculating layout physics every frame. The graph publishes non-sensitive DOM state for zoom, screen bounds, and internal edge-vertex counts so validation can verify that the full clustered graph fits inside the viewport and that bend-marker coverage is active. Tag and cluster inspector actions run analysis directly when tag-cloud.html is opened through npm run workdb -- serve; static file mode falls back to the CLI command. File/project/session inspector actions can build a context pack, preview safe markdown/text content, and reveal indexed paths in Finder from localhost only. Each analysis writes a markdown dossier plus a JSON manifest with all matched files, projects, sessions, and related tags.

Local server API:
- GET /api/search?q=<query>&limit=20 returns ranked project/file/session matches.
- GET /api/context?q=<query>&limit=12 returns a markdown context pack; add format=json for JSON wrapping.
- GET /api/file?id=<graph-id-or-path> returns metadata and safe preview content for indexed non-sensitive text files.
- POST /api/open with id/path/graphId reveals an indexed path in Finder. This action is restricted to localhost requests.
- POST /api/analyze runs tag or cluster analysis and writes analysis outputs.

Agent usage:
- Use npm run workdb:cloud -- "<topic>" --limit 12 when Codex should use the live remote Firestore DB as a remote-safe context pack.
- Before working on an unfamiliar project or topic, run npm run workdb -- context "<topic>" --limit 12.
- Use npm run workdb -- search "<topic>" --json for ranked machine-readable routing.
- Use npm run workdb -- show "<graph-id-or-path>" only after choosing a specific evidence item.
- Treat workdb as a routing/context index, then inspect authoritative repo files before editing.

Public snapshot:
- assets/tag-cloud-snapshot.json is generated from the private graph but contains only coordinates, colors, node types, radii, opaque stable node keys, and edge indexes.
- It intentionally excludes labels, titles, local paths, remotes, snippets, source IDs, and file names.
- The landing page uses this snapshot as a label-free first-viewport graph layer.

Remote Work DB context:
- outputs/global-work-kb/remote-workdb-context.json is generated from outputs/global-work-kb/db.json.
- The authenticated app reads this data from vaults/main/workdbContext after Google sign-in.
- It contains summary, cluster, project, tag, and external inventory cards with counts, safe tags, cluster labels, safety flags, and local Codex follow-up commands.
- Each remote record can include a cloud context command for querying the live Firestore layer and local commands for exact private drill-down.
- It intentionally excludes exact local paths, file snippets, file content, and git remotes. The local CLI remains the source for exact path reveal and file preview.
- The authenticated app opens on Work DB by default, renders the label-free Work DB tag cloud snapshot only on the Work DB view, and presents that snapshot as a standalone graph explorer panel rather than a passive top-of-screen image. The graph explorer owns its header, compact Map/Agent lens/Expand toolbar, canvas viewport stage, expand/dock mode, multitouch pinch zoom, mouse-wheel zoom, double-click zoom, drag-pan behavior, and a Map recenter action for touch recovery. It maps clickable graph nodes to remote-safe Work DB records at runtime through opaque stable node keys with ordinal fallback, shows cloud status cards for Firestore connectivity, privacy mode, and Codex context commands, and fails clearly if workdbContext is empty or missing its summary document.
- The authenticated graph renderer uses zoom-aware level-of-detail filtering. At far zoom levels it draws only major memory/cluster/project/tag landmarks and core links to avoid a visual knot and reduce canvas work. As the user zooms in, additional node classes, smaller radii, and more links are progressively admitted. Runtime DOM dataset markers expose current zoom, visible node count, visible edge count, and total nodes so smoke/visual QA can verify that the graph is not rendering the full private snapshot at the most distant view.

Safety rules:
- Secret-looking files are indexed by path, size, mtime, and sensitive flag only.
- JSON/YAML/TOML/config-like files are indexed by metadata only, not snippet text.
- Snippets redact API-key-like values, token-like fields, and email addresses before writing files.jsonl.
- Generated dependency/build folders such as node_modules, .git, dist, build, coverage, .firebase, cache, worktrees, and DerivedData are skipped.
- outputs/global-work-kb is the private deliverable; the public repository should contain only the generator code and documentation.
- raw-registry.jsonl and chronology.jsonl may include private paths and file titles, so keep them local/private with the rest of outputs/global-work-kb.
- Git remotes are redacted for URL userinfo/token patterns before they are written into project, catalog, or graph outputs.

Runtime
- No application server is required.
- No frontend package installation is required.
- The public site can be opened directly from index.html or served by any static host.
- app.html should be served from Firebase Hosting for Google sign-in and Firestore reads. Direct file opening is only for layout review.

Authentication Model
- The workspace uses Firebase Authentication with Google sign-in.
- The frontend also checks allowedEmails in firebase-config.js for UX clarity.
- The real access boundary is Firestore rules, which allow read access only for request.auth.token.email == "j3d1fm@gmail.com" and request.auth.token.email_verified == true.
- Client writes are disabled by rules.
- Private source imports should not be shipped in dist/ or committed to a public repository.
- The public sign-in screen intentionally avoids naming the backend stack, allowlist, or account restriction details; these details belong in rules, scripts, and operator documentation.
